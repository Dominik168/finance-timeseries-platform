Co stavím a proč

Tenhle projekt je finance time-series data platforma postavená na Azure Databricks — denní OHLCV data pro microcap AI akcie (vč. mého sledovaného HYFT), s referenčními tickery SPY, QQQ, SMH a VIX jako market-wide volatility signálem.

Cíl je dvojí:

  1. Portfolio piece pro pohovory na pozice Data/ML Engineer se zaměřením na time-series a finance/quant prostředí
  2. Reálný nástroj — dashboard, který budu sám používat k monitoringu watchlistu

Architektura je postavená jako medallion (Bronze → Silver → Gold), celé jako config-driven framework (ne ad-hoc skripty) — přidání nového tickeru nebo feature bude znamenat úpravu YAML configu, ne psaní nového kódu. Tech stack: Databricks Asset Bundles, Unity Catalog, PySpark, MLflow, a později Lakebase + Databricks Apps pro serving a dashboard.

Proč finance/microcap a ne třeba neuroscience (můj původní nápad) nebo energetika? Protože mám reálný osobní zájem sledovat tyhle akcie, microcapy mají dramatičtější regime changes než blue-chipy (vizuálně i analyticky bohatší data), a finance bakalář mi dává přímou relevanci pro CV. Productive struggle funguje nejlíp, když člověku na výsledku skutečně záleží.

Den 1-2: Infrastruktura

Než jsem napsal jediný řádek Pythonu, bylo potřeba postavit základ:

  • Resource group + budget alerty (30/50/80/100 %) — s $200 startovním kreditem je tohle první věc, kterou každý seriózní cloud projekt potřebuje
  • Automation Account s managed identity jako pojistka pro automatické zastavení resources, kdyby alerty nestačily
  • Azure Databricks workspace (Premium tier) — Premium je nutnost, protože jen ten podporuje Unity Catalog
  • Databricks CLI (nový, Go-based, ne starý databricks-cli pip balíček — to jsou dva different nástroje, jen ten novější umí bundle init)
  • GitHub repo + Databricks Asset Bundles (DAB) inicializace

Klíčové poznání: Unity Catalog je dnes default

Čekal jsem, že budu muset ručně zakládat metastore přes Databricks Account Console. Nakonec to nebylo potřeba — Unity Catalog je automaticky aktivní pro každý workspace vytvořený po listopadu 2023. Stačilo otevřít Catalog panel ve workspace a byl tam.

To je dobrá připomínka, že dokumentace a tutoriály na internetu rychle zastarávají — co byl loni nutný manuální krok, dnes platforma řeší sama.

Den 3: DAB konfigurace, cluster a separace dev/prod

Proč jeden workspace, ne dva

Zvažoval jsem, jestli pro dev a prod nepotřebuju dva oddělené Databricks workspace. Rozhodnutí: ne — jeden workspace stačí, separaci řeším přes Unity Catalog catalogy (finance_dev vs finance_prod). Pro solo portfolio projekt by druhý workspace jen zdvojnásobil náklady bez přidané hodnoty. Tohle je vědomá architektonická volba, ne zjednodušení — a přesně takhle to budu umět obhájit na pohovoru.

V databricks.yml mám teď dva targets:

targets:
  dev:
    mode: development
    variables:
      catalog: finance_dev
  prod:
    mode: production
    variables:
      catalog: finance_prod
    permissions:
      - user_name: azuredominik007@seznam.cz
        level: CAN_MANAGE

Zajímavý detail: v dev módu Databricks automaticky prefixuje všechny resources ([dev username]) a tebe nastaví jako vlastníka, takže explicitní permissions blok tam nepotřebuješ. V prod módu je to naopak — nic není implicitní, takže přístup musíš nastavit ručně.

Interaktivní cluster vs. job cluster

Další důležité rozlišení, které jsem si ujasnil: interaktivní cluster (pro vývoj v notebooku) se vytváří manuálně v UI a běží, dokud ho nezastavíš. Job cluster je naopak definovaný přímo v DAB bundle, vytvoří se automaticky při spuštění jobu a po doběhnutí zanikne — to bude standard pro produkční pipeline (bronze→silver→gold). Dnes jsem si založil jen ten první, pro vývoj.
