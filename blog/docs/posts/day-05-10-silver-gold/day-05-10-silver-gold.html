What we built

Two full layers of the medallion architecture are now complete. Silver transforms raw OHLCV data into a rich feature set ready for machine learning. Gold introduces a proper star schema on top of it — dim_date, dim_ticker with SCD Type 2, and an insert-only fact table.

Silver layer (Days 5-7): feature engineering

The Silver layer is where raw price data becomes useful. Everything runs through features.py, a config-driven module that reads features.yaml — adding a new feature means editing the YAML, not touching the Python logic.

Cleaning: the illiquid day flag

Microcap stocks like HYFT regularly have days with zero trading volume — the stock simply didn’t trade. Rather than dropping these rows (which would break rolling window continuity), we flag them with an is_illiquid boolean column and keep them in the data.

One subtlety worth noting: ^VIX also has zero volume on every single row, but for a completely different reason — it’s an index, not a traded security, so the concept of “trading volume” doesn’t apply. We maintain a volume_exempt_tickers list in features.yaml to handle this case without special-casing it in the code.

Log returns

Rather than working with raw prices, we compute log returns: ln(price_today / price_yesterday). Two reasons:

Stationarity — raw prices trend up or down over time, which violates assumptions of most statistical and ML models. Log returns fluctuate around zero regardless of whether the stock is at $5 or $500.

Time additivity — log returns can be summed across time periods. Simple percentage returns cannot: a +100% gain followed by a -50% loss nets to zero, but 50% ≠ 0. Log returns handle this correctly.

The first row per ticker always has log_return = NULL — there’s no previous day to compare against. This is expected and correct.

Rolling volatility (20d and 60d)

Rolling volatility is the standard deviation of log returns over a backward-looking window. “Backward-looking” is the critical constraint: the window can only use past data, never future. Using future data here would be a look-ahead bias bug — the model would appear to predict the future because it was trained on it.

Both windows use Spark’s rowsBetween(-(N-1), 0) pattern, which is the clean way to express “current row plus the N-1 rows before it” in a window function. The 20-day window reacts quickly to recent volatility spikes; the 60-day window gives a smoother, longer-term picture. Both will feed into the clustering model as distinct signals.

We hit a Photon engine bug with 60-day windows that caused a RuntimeError: Incorrect sliding frame boundaries. The fix was straightforward — spark.conf.set("spark.databricks.photon.enabled", "false") disables the Photon accelerator for that computation. This is a known issue in certain Databricks runtime versions with larger sliding frames.

Moving averages (20, 50, 200)

Same rowsBetween window pattern, but with F.avg() instead of F.stddev(), applied to adj_close rather than log_return. The three windows correspond to roughly one month, one quarter, and one year of trading days — the standard technical analysis trio. Whether current price is above or below each moving average is a classic regime signal.

RSI

RSI (Relative Strength Index) is a momentum oscillator on a 0-100 scale. The calculation splits daily price changes into gains and losses, averages each over a 14-day window, and computes 100 - (100 / (1 + avg_gain/avg_loss)). Values above 70 are conventionally “overbought”, below 30 is “oversold”.

The one edge case worth handling: if avg_loss = 0 (all 14 days were gains), the formula would divide by zero. We return 100.0 in that case — pure-gain streaks saturate RSI at its maximum.

VIX correlation (60d)

VIX measures the market’s expected volatility over the next 30 days, derived from S&P 500 options prices. It’s commonly called the “fear index” — when markets panic, VIX spikes.

For our microcap universe, VIX correlation tells us whether a stock moves with market-wide sentiment or independently. A positive correlation means the stock tends to fall when fear rises (typical for speculative microcaps). A negative correlation is unusual — it suggests the stock reacts to company-specific news rather than macro sentiment.

Computing this required a self-join: since all tickers share one Silver table, we had to extract VIX returns as a separate DataFrame and join them back by trade_date before running the rolling F.corr() window.

NaN cleanup

After running all transformations, we found two rows with NaN (Not a Number) values — both for the current trading day, where adj_close wasn’t yet finalized by yfinance. NaN is different from NULL in Spark: isNull() won’t catch it, and it propagates silently through calculations. A final clean_nan() pass replaces all NaN values with NULL before writing to the Silver table.

Validation and tests (Day 8)

Before moving on, we validated the Silver table:

  • No NULL values in any column (confirmed — the sparse history for newer microcaps is just shorter, not missing)
  • Zero volume days correctly flagged as illiquid, VIX correctly exempt
  • NaN count across all feature columns: 0 after cleanup

We then wrote 10 pytest unit tests covering add_illiquid_flag(), add_log_returns(), and clean_nan(). Tests run locally against a local[1] SparkSession — no cluster connection needed, which keeps the feedback loop fast.

One thing worth noting about the test structure: the FeaturesConfig in tests is hardcoded, not read from features.yaml. This is intentional. Unit tests test isolated function logic, not the full pipeline including config loading. If a test read the YAML, changing a config value would break tests even when the function itself is perfectly correct — a false alarm.

Gold layer (Day 10): star schema

The Gold layer introduces dimensional modeling on top of Silver. Three tables, each with a specific role.

dim_date

A pre-generated calendar table covering 2015-2030, built once and never updated (the calendar doesn’t change). It’s a conformed dimension — shared across every fact table in the Gold layer, so that “quarter” or “month” always means exactly the same thing regardless of which fact table you’re querying.

The surrogate key is date_sk as an integer in YYYYMMDD format — a standard Kimball convention that makes date arithmetic readable without needing to join back to the dimension.

dim_ticker (SCD Type 2)

This dimension tracks when each ticker entered our tracked universe, and — as the SCD Type 2 tracked attribute — what market_cap_category it currently belongs to.

On each pipeline run, update_dim_ticker_scd2() compares the current category (derived from the latest valid closing price) against what’s stored. If it changed, the old row gets closed (effective_to = today, is_current = False) and a new current row gets inserted with a fresh surrogate key.

One implementation detail worth being explicit about: the “market cap category” here is derived from price alone, not from actual market cap (price × shares outstanding). This is a deliberate simplification — yfinance.history() doesn’t return shares outstanding, so we use price thresholds as a proxy. In a production system, you’d pull shares outstanding from a separate API endpoint. The SCD2 mechanics are identical regardless.

We also handle late-arriving dimensions via add_inferred_members(): if a fact row references a ticker that doesn’t yet exist in dim_ticker, a skeleton row gets inserted automatically rather than letting the fact become an orphan FK.

fact_ohlcv (insert-only)

The fact table at ticker × day grain. It joins to both dimensions via surrogate keys (ticker_sk, date_sk) and carries all Silver features as measures. Insert-only means once a trading day is recorded, it never changes — there’s no SCD logic on a fact table, only on dimensions.

After building the fact table, check_referential_integrity() runs a left_anti join to verify that every foreign key in the fact table has a matching row in its dimension. Both orphan counts came back at 0.
